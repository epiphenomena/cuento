# DECISIONS.md — cuento

Why the design is the way it is. AGENTS.md defines *how* to work; PLAN.md defines *what* to build, in order; this file records *why*. When they conflict, this file wins for rationale.

Seeded at p00.1 from the settled-decisions table in PLAN.md. Later steps append one line per non-trivial choice (see AGENTS "Ambiguity"). Entries are append-only in spirit: supersede rather than silently rewrite.

## Settled decisions (D1–D26)

| ID | Decision | Rationale |
|----|----------|-----------|
| D1 | Amounts stored as `int64` in the currency's **minor units**; exponent per currency from `currencies` table. | Exact arithmetic; JPY/BHD-style exponents cost nothing to support. |
| D2 | **Net-debit** signed amounts (debits +, credits −). Every transaction sums to exactly 0 in its currency — and to 0 **within each fund** (D20). Signed vs DR/CR is display-only, per user. | Single column; mixed revenue/expense subtrees subtotal naturally. |
| D3 | Transactions are **single-currency**. Cross-currency flows = two transactions through a multicurrency **FX Clearing** account (equity-class). Its converted balance on reports is cumulative FX gain/loss. GnuCash-style per-split value/amount pair rejected for schema+UI complexity; revisit if clearing proves awkward. | Keeps the zero-sum invariant trivial and honest. |
| D4 | Audit = append-only `*_versions` tables: entity id + `change_id` + `valid_from` + `op(create/update/delete)` + full snapshot of business columns. Current tables are the denormalized latest state. State as of time T = row with max(`valid_from`) ≤ T per entity, excluded if `op='delete'`. Versions tables never see UPDATE/DELETE. | Fast current-state queries, mechanical point-in-time, pure append. |
| D5 | v1 audit surface: per-transaction history UI + per-entity as-of store queries + integrity check that current == latest version. Whole-report "books as edited at time T" is supported by the data but deferred as a feature. | Keeps the report toolkit single-plane for v1. |
| D6 | **sqlc** (queries) + **goose** (migrations, embedded). Forward-only migrations; runner backs up the db file before applying. No down migrations. | SQL stays visible/optimizable; backup beats theoretical rollbacks. |
| D7 | Driver: `modernc.org/sqlite` (pure Go). | CGO-free static binary, trivial cross-compile; perf ample at this scale. |
| D8 | Hosting: GCE **e2-micro always-free VM** + persistent disk + **Litestream → GCS** (free 5 GB). TLS via in-process autocert. Cloud Run rejected: ephemeral FS + scale-to-zero vs a single-writer SQLite file. | One binary, one VM, one systemd unit. |
| D9 | Auth primitives: scs sessions (SQLite store), argon2id, stdlib `http.CrossOriginProtection` (fallback nosurf), `html/template` escaping + strict CSP, `x/time/rate` on login. | Nothing hand-rolled; everything provable via route-registry tests. |
| D10 | Permissions: per-user `txn_perm ∈ {none,read,write}` + per-report-group read grants; `is_admin` implies everything. Report groups are declared in code and synced to the db at startup. Permissions are org-global in v1 (per-subsidiary grants → backlog, Q2). | Matches stated needs; no RBAC machinery. |
| D11 | Tree rules: A/L/E children must match parent type exactly; revenue and expense may interleave freely under R/E parents. Placeholders (accounts with children) hold no splits. | Balance sheet stays clean; mixed groupings stay possible — the chart holds *natural* categories, since programs are a dimension (D24). |
| D12 | FX conversion happens only in reports: balance-sheet figures at the closing (as-of) rate; P&L activity at each transaction-date rate; rate lookup = latest on-or-before date, direct pair then reciprocal. Rates stored as REAL; conversion rounds half-even at final aggregates. | Standard treatment; storage stays integer-exact. |
| D13 | Splits in a **finalized** reconciliation are locked (amount/account/txn/date/fund); editing requires an audited unreconcile first. Reconciliation is per `(account, currency)` and spans all funds — a bank statement covers one balance. | Statements stay provable; a key reason funds are a dimension, not currencies (D20). |
| D14 | **Bilingual UI (en, es) in v1**, extensible: `internal/i18n` embedded key→string catalogs + `t` template func; missing key falls back to en; a parity test enforces identical key sets across catalogs; adding a language = adding one catalog file. No CLDR or i18n dependency. Supersedes the earlier English-chrome-only decision. | The bookkeepers work in Spanish; catalogs are a page of code, not a framework. |
| D15 | Dependency allowlist: modernc.org/sqlite, pressly/goose/v3, sqlc (tool), alexedwards/scs/v2 (+sqlite3store), alexedwards/argon2id, golang.org/x/crypto (autocert), golang.org/x/time, google/go-cmp (tests). Additions need a DECISIONS entry *and* explicit human acknowledgment (AGENTS rule 1). | Lightweight by construction. |
| D16 | Date/number formats are small enumerations implemented and tested in `internal/money` — no CLDR dependency. Date entry uses text inputs (never `input[type=date]`). | Users pick from a handful of formats; browsers don't get a vote. |
| D17 | IDs are `INTEGER PRIMARY KEY AUTOINCREMENT` on financial tables (no rowid reuse after delete — audit-friendly). | Cheap referential permanence. |
| D18 | **Subsidiaries**: `subsidiaries` is a tree with exactly one root; each has a `base_currency`. Accounts map to **one or more** subsidiaries via `account_subsidiaries`, with the invariant **parent set ⊇ union of children's sets** (superset; assigning a sub auto-propagates to ancestors). Every transaction belongs to **exactly one** subsidiary and all its split accounts must include it. Report scope = a chosen subsidiary **consolidated with all descendants**; the root is full consolidation. Default report currency = the scoped subsidiary's base currency. | NetSuite-shaped without OneWorld machinery. |
| D19 | **Intercompany**: cross-subsidiary funding = paired transactions through due-to/due-from accounts flagged `intercompany`. Consolidated reports whose scope covers both sides collapse flagged accounts after asserting they net to zero per currency; a nonzero net renders as a warning row. Elimination journal entries are out of scope. | The flag makes consolidation honest and cheap. |
| D20 | **Restricted funds are a split dimension with fund-level conservation.** `funds` documents grants; a fund scopes to **one or more subsidiaries** via `fund_subsidiaries` (not inherited) and optionally to a **program subtree**. Every split carries `fund_id` (NULL = unrestricted); every transaction must sum to zero **within each fund**. Sub-currency modelling rejected: identical semantics but breaks single-statement reconciliation (D13), multiplies the currency/rate surface, and cross-currency conversion would mint pseudo-currencies. GAAP "released from restrictions" is **derived** in reporting, not journaled. | All the tracking power of the sub-currency idea, none of the collateral damage. |
| D21 | **Functional expenses (990 Part IX)** are a fixed enum `program | management | fundraising`. Expense accounts carry a **default** class; every expense split **requires** a class (prefilled, overridable); non-expense splits must be NULL. Trigger-enforced. Part IX is natural × functional — a matrix, so a dimension, not a duplicated hierarchy. | One report renders Part IX; the tree stays single and clean. |
| D22 | Import source is the **cleaned full-ledger CSV**. `cmd/ledgerimport` is driven by a reviewable mapping file: per source account → {type, parent, subsidiaries, default functional class}; per row → subsidiary and fund (defaults: root subsidiary, unrestricted) with optional source-column overrides. | The export is already a ledger; converting it beats re-keying history. |
| D23 | Negative restricted-fund balances (spending a grant past its receipts) are surfaced as **warnings** by `cuento check` and on fund pages — never blocked at write time. | Backdated entries make hard blocks hostile; visibility beats prevention. |
| D24 | **Programs are a dimension**: `programs` tree with a single seeded root ("General"); every **revenue and expense** split carries `program_id` (required; prefilled from the account default, else root); A/L/E splits carry none. Programs are org-global (Q5). The chart holds *natural* categories; mission structure lives in the program tree. Program and functional class are **orthogonal**. | 990 needs program-level revenue and expenses; duplicating programs in the tree was the same mistake as functional duplication. |
| D25 | **990 line mapping**: seeded `form990_lines` reference table (Parts VIII, IX, X); `accounts.form990_code` nullable with **effective code = own or nearest ancestor's** (inheritable). Not hard-required: 990 reports render an explicit **Unmapped** bucket instead of dropping rows; a warning check flags active R/E leaves with activity but no effective code. | One field, set ~10 times, and year-end becomes running four reports. |
| D26 | **Two-step go-live**: the production db is produced by `cmd/ledgerimport` from the cleaned CSVs + a reviewed mapping file, rehearsed during p09.4 (locally; mapping gitignored beside the source data). Deploy = (1) build db + `cuento check --strict`, (2) ship db + binary. The historical mapping assigns subsidiaries, funds, programs, functional classes, and 990 codes up front. | Import is the deployment path, not an afterthought. |

## Implementation notes (appended per step)

- **p00.2** — `make lint` runs golangci-lint's bundled gofumpt formatter *and* the standalone `gofumpt` binary; the two disagreed on import grouping (standalone splits the module-local `cuento/...` import into its own block, the bundled one merged it). Set `formatters.settings.gofumpt.module-path: cuento` in `.golangci.yml` so both produce the split form. Tooling reconciliation only; no design impact.
- **p00.3** — Pinned the Go toolchain to **1.25.12** (`go 1.25.12` in `go.mod`; CI uses `go-version-file: go.mod`). govulncheck flagged 9 stdlib vulnerabilities (crypto/tls, crypto/x509, net, net/url, os, net/textproto) present in the locally-installed 1.25.7 and all fixed by ≤1.25.12; `GOTOOLCHAIN=auto` auto-downloads it. Keeps the vuln scan clean without touching any allowlisted dependency (D15). Bump the pin as later patches land.
- **p00.3 (DoD clarification, per human 2026-07-11)** — There is no hosted CI and the human manages pushes themselves. The definition of "green on the repo host" for **every** step in this project reduces to: `make lint test check` (plus the CI workflow's steps: `go vet`, `golangci-lint`, `gofumpt -l`, `go test ./...`, `govulncheck ./...`) all pass **locally**. The `.github/workflows/ci.yml` file is committed for whenever a remote is added, but a step is "done" once locally green.
- **p04.1** — Versioned tables seed with full audit consistency: a migration that seeds a versioned business row also inserts its `changes` row (actor = system user 1, fixed epoch timestamp `'1970-01-01T00:00:00Z'`, `kind='seed'`) and its `*_versions` `op='create'` row (`valid_from == changes.at`), so Z3/Z5 hold uniformly with no seed special-casing. The `subsidiaries_versions` snapshot column set (`parent_id, name, base_currency, active, sort_order`) is documented in the migration; p04.2's `InsertSubsidiaryVersion` query must write exactly those. Single-root is enforced by a **pair** of triggers (BEFORE INSERT + BEFORE UPDATE): on INSERT the AUTOINCREMENT `NEW.id` is unassigned, so the guard checks plain existence of another root (an `id <> NEW.id` clause would compare against NULL and never fire); on UPDATE it excludes the row with `id <> NEW.id`. The **program root (p07.1)** gets the same seed-with-version + single-root-trigger-pair treatment. **Deferred instance:** the system user (seeded in 00002 before `users_versions` existed) has NO version row yet — when **p06.1** creates `users_versions`, that migration MUST backfill a `op='create'` version row for user id 1 (excluding `password_hash`, rule 5) or Z3 will flag `users`.
- **p04.2** — Version-append convention established concretely (the pattern p05/p07/p08 copy): inside the funnel's `fn`, the entity op does its LIVE write FIRST, then appends the version row via a **snapshot-from-live** query (`InsertSubsidiaryVersion` = `INSERT ... SELECT` from the just-written live row), so the version row is byte-identical to the live row (Z3 cannot diverge) and `valid_from == changes.at` **by construction** — both structural, never asserted. **All validation lives inside `fn`** (cycle/root/active-children checks read the tx-bound `*sqlc.Queries`), so a rejected op returns its error out of `fn` and the funnel rolls the change row back — rejected ops leave **no audit trace** (asserted by the blocked-op no-trace tests). Deactivate is `op='update'` (the entity persists), never `'delete'`. Op mapping: create→`create`, rename/move/base-currency/deactivate→`update`. **Tooling quirk:** sqlc v1.31.1 **miscounts byte offsets** when a query `.sql` file contains multi-byte UTF-8 (e.g. em-dashes in comments), silently corrupting the generated SQL constants for the *whole* file (not just the offending query). Keep every `.sql` query file **pure ASCII**. Also: sqlc's sqlite parser rejects `sqlc.arg()`/`@name` and mangles numbered `?N` params inside an `INSERT ... SELECT ... WHERE` form, so `InsertSubsidiaryVersion` uses plain positional `?` (each param used once: op, change_id, entity_id) — the store wraps the resulting `Op/ID/ID_2` field names behind one `insertSubsidiaryVersion` helper.
- **p05.2** — Account ops copy p04.2's funnel/version discipline with three deliberate departures. (1) **Composite-key versions:** `account_names`/`account_subsidiaries` version rows key on `(account_id, lang)` / `(account_id, subsidiary_id)`; their `InsertAccountNameVersion`/`InsertAccountSubsidiaryVersion` copy `InsertSubsidiaryVersion`'s snapshot-from-live shape, changing only the `entity_id`/WHERE. `testutil.AssertVersionedName`/`AssertVersionedSub` filter on the composite key (the single-id `AssertVersioned` stays for `accounts`). (2) **Removal inverts live-write-first:** a membership removal is `op='delete'` whose snapshot is the *last-known* membership, so `removeSub` inserts the version row **before** deleting the live row (an add keeps the normal live-first order). (3) **Subsidiary-set asymmetry (D18):** `parent set ⊇ union of children's` is over the **account tree**, not the subsidiary tree — assigning subsidiary `A1` to an account does **not** add `A1`'s subsidiary-tree ancestors; report scoping consolidates descendants at read time (D18). **Create propagates, move rejects (resolved asymmetry):** `CreateAccount`/`SetAccountSubsidiaries` **auto-propagate** an assigned sub UP the account-ancestor chain (each ancestor missing it gains a membership + `op='create'` version; an ancestor already holding it is a no-op with no version row); `UpdateAccount` moves **reject** with `ErrSubMismatch` when the new parent's set does not cover the mover's. Removing a sub is blocked while a **direct child** still maps it (`ErrSubInUseByChild`); the split-usage half of that guard is **deferred to p08** (no `splits` table yet) as a tagged `// TODO(p08)` + a documented test note. **Functional-class-on-non-expense** is rejected cleanly with a typed `ErrFunctionalClassNotExpense` before the trigger fires (Build text said "validate cleanly" but named no error). **990 type check** matches the account's `type` against `form990_lines.account_types` by **CSV membership** (some lines allow >1 type), not equality (`Err990TypeMismatch`). **`Effective990Codes` resolves the nearest-ancestor code IN GO** from a flat `(id, parent_id, own_code)` fetch: sqlc v1.31.1's sqlite analyzer cannot resolve a *carried* recursive-CTE column in the output projection (it reports the carried column "does not exist"), so a value-carrying downward CTE is not expressible — the in-memory walk over the small chart is the store-wrapped alternative.
- **p06.1** — Credentials/perms/settings on `users` + the auth primitive. **The invariant (rule 5):** `users_versions` has **no `password_hash` column** and `InsertUserVersion` (snapshot-from-live) selects every users column **except** it, while `CreateUser`'s live `InsertUser` **does** write the hash — that asymmetry is the whole step, proven by `TestUsersVersionOmitsPasswordHash` (PRAGMA table_info has no such column; a `SELECT *` scan of the latest version row finds the secret in no column; the live row does hold it). **System-user version backfill (the p04.1-deferred instance):** 00006 inserts `changes` id **2** (the only prior change is id 1, 00004's root-subsidiary seed; 00003/00005 seed none) and a `users_versions` `op='create'` row for user id 1 via `INSERT...SELECT` from the just-ALTERed live row (so the snapshot equals the post-ALTER defaults by construction and cannot drift), keeping Z3/Z5 uniform — without any hash. `AssertVersioned` needed no change (single-id entity). **`report_groups`** is code-synced reference data (D10), **not** versioned (no twin) and **unseeded** here — the startup sync fills it in p06.3. **`user_report_grants`** + its composite version twin (`entity_id=user_id`, snapshot `group_name`, like `account_subsidiaries`) are schema-only this step: grant writers/version-append land in p13.2, so grant FKs are exercised by **direct SQL** tests, not a store method. **argon2id (`internal/auth`, D9/D15, the ONLY place it is called):** `Hash`/`Verify` wrap `argon2id.DefaultParams` (v1.0.0): Memory 65536 KiB (64 MiB), Iterations 1, **Parallelism = `runtime.NumCPU()`** (machine-dependent — encoded into each PHC string, so verification is self-describing; a fixed value for reproducible cross-host cost is a one-line future tuning), SaltLength 16, KeyLength 32. **`go get github.com/alexedwards/argon2id`** (allowlisted D15, no extra ack needed) is the only new **direct** dep; it transitively pulls `golang.org/x/crypto` (also D15) as **indirect**. That brings govulncheck finding **GO-2026-5932** (`golang.org/x/crypto/openpgp` unmaintained; Fixed-in: N/A) as an **uncalled/informational** finding — govulncheck exits 0, "code affected by 0 vulnerabilities" — because cuento never imports that subpackage; it does **not** block the DoD. **sqlc regen note:** after 00006 widens `users`, `GetUser`'s explicit-column projection makes sqlc emit a bespoke `GetUserRow` (was `sqlc.User`); the fields it scans are unchanged so the existing `sqlc/users_changes_test.go` still compiles. `InsertUserVersion` uses the `INSERT...SELECT...WHERE` positional-`?` form (generated `Op/ID/ID_2`), wrapped behind `insertUserVersion` like the subsidiary/account helpers; `users.sql` kept pure ASCII.
- **p05.3** — `Tree`'s name resolution became a **COALESCE fallback chain requested-lang → en → any** (D14), replacing the p05.2 `LEFT JOIN ... COALESCE(an.name,'')` in **both** `AccountTree` and `AccountTreeBySubsidiary` (updating only one is the obvious latent bug; the test exercises both the nil-filter and subFilter paths). The chain is three **correlated subqueries** on `account_names` (requested lang, then `'en'`, then `ORDER BY lang LIMIT 1` for the deterministic "any"), with a trailing `''` so an account with no names at all still resolves to a value. Two sqlc v1.31.1 sqlite-analyzer quirks shaped the final SQL: (1) an unaliased `account_names` inside a correlated subquery that also references the outer `tree.id` makes the analyzer flag `lang`/`name` as **"column reference ambiguous"** — each subquery gets its own table alias (`an1`/`an2`/`anx`) with every column qualified; (2) a bare `COALESCE(<subqueries>, '')` is typed by sqlc as `interface{}` (not `string`), which would break the `TreeRow.Name string` mapping, so the whole expression is wrapped in `CAST(... AS TEXT)` to pin the generated column to `string`. No store-Go change was needed (the mapping is unchanged); the `AccountTreeBySubsidiaryParams` fields (`Lang`, `SubsidiaryID`) kept their names despite the param **order** flipping (lang now precedes subsidiary in the SQL), and the store uses named-field structs so the reorder is invisible. **Test-data deviation:** the "any" case needs an account with *neither* the requested lang *nor* `en`, but `CreateAccount` mandates an `en` name and there is no remove-name API — so the test raw-`DELETE`s the seeded `en` row from `account_names` after creation (raw SQL in store tests is already in-convention, e.g. `nameAsOf`). The task's "build with CreateAccount/SetAccountName" cannot otherwise reach the any-branch.
- **p06.2** — Web sessions + login + the security middleware spine every later web step inherits (D9, rule 13). **`Handler` signature changed** to `Handler(cfg web.Config, db *sql.DB, st *store.Store)` (Config carries `Version` + `Dev`); the p00.2 `Handler(version)` callers/tests were updated. `serve` now `db.Open`s after migrating (it needs the `*sql.DB` for both the store and the scs store) and gained a **`-dev`** flag (the Makefile `run` target already passed it). **Middleware chain (outer→inner, so headers wrap even deep errors and cross-origin rejects before any work):** `secureHeaders → crossOrigin (stdlib http.CrossOriginProtection) → scs LoadAndSave → auth (resolve current user from session id, re-read each request so perm/locale changes need no re-login; missing/disabled ⇒ anonymous, not error) → lang (user locale → ?lang query → lang cookie → en)`. **scs session table:** created by goose migration **00007** (`sessions(token TEXT PK, data BLOB NOT NULL, expiry REAL NOT NULL)` + `sessions_expiry_idx`) so goose stays canonical (rule 4) — NOT scs auto-create. Schema **verified against the installed library source** `github.com/alexedwards/scs/sqlite3store@v0.0.0-20251002162104-209de6e426de` (its `Find`/`Commit`/`Delete` raw SQL uses `token`/`data`/`expiry` with `expiry` a `julianday()` REAL; default table name `sessions`); matches the library README DDL exactly. `sessions` is scs-owned infra, **not** a versioned business table (no `_versions` twin, no `changes` wiring, no seed). **Cookie:** name `cuento_session`, `HttpOnly`, `SameSite=Lax`, `Path="/"`, `Secure = !Dev`, 12h Lifetime / 2h IdleTimeout. **Uniform auth error / no enumeration (rule 13):** unknown user, wrong password, AND disabled account all return the **same** 401 + identical body (`auth.invalid`). Timing is equalized too, not just the body: a fixed **decoy argon2id hash** (built once at startup via `auth.Hash`) is verified-and-discarded on the unknown-user / passwordless / DB-error paths so every attempt spends one argon2 verify regardless of whether the username exists. On success: `RenewToken` (session-fixation defense) then `Put(user_id)`. **CSP** (strict, no inline script, rule 12): `default-src 'self'; script-src 'self'; style-src 'self'; img-src 'self'; connect-src 'self'; font-src 'self'; object-src 'none'; base-uri 'none'; frame-ancestors 'none'; form-action 'self'` — plus `X-Content-Type-Options: nosniff`, `Referrer-Policy: same-origin`, `Cross-Origin-Opener-Policy: same-origin`, `X-Frame-Options: DENY`. HTML `no-store` is set by the HTML render path (`render`), NOT the header middleware, so `/static` + `/healthz` keep their own caching (p10 owns asset headers). **Rate limit** (`x/time/rate`): keyed by **IP+username**, burst 5, refill 1 token/30s, checked **before** any argon2 work (so throttling actually saves the hashing cost); over-limit ⇒ **429** with the login page + `auth.rate_limited`. Limiter map is bounded (cleared wholesale at 4096 keys — the simplest bound that only ever loosens, never locks out). Client IP = direct `RemoteAddr` only (never a spoofable forwarded header). **`{{t}}` per-request localization:** templates parse once with a stub `t`; `render` clones the set and rebinds `t` to a closure over the resolved lang before Execute (html/template's func map is what carries per-request state), so `?lang=es` renders the Spanish catalog. **New login keys** (`auth.login_title/username/password/sign_in/invalid/rate_limited`) added to BOTH catalogs (parity test stays green). **"Subsequent request authenticated" tested without p06.3 routes:** `GET /login` redirects (303) when identity resolves, so the success test asserts the cookie authenticates without mounting a throwaway route that p06.3's registry-completeness test would later reject. **Store reads added** (rule 2 permits reads via sqlc): `CredentialsByUsername` (`UserByUsername` query) and `UserByID` — both NEW queries; `GetUser` untouched (its projection is pinned by `sqlc/users_changes_test.go`). **Deps:** `scs/v2`, `scs/sqlite3store`, `x/time` added (all D15-allowlisted, no extra ack); `go mod tidy` flipped them direct. No non-allowlisted direct dep (sqlite3store's own test-only `mattn/go-sqlite3` is a go.sum build-list entry, not in our require graph). govulncheck exit 0 (the single uncalled `x/crypto/openpgp` GO-2026-5932 finding from p06.1 persists, informational). All `.sql`/migration files kept pure ASCII (p04.2 quirk).
